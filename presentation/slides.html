Security

17 podatności — najważniejsze

Krytyczne CRITICAL

  • SQL Injection w 6+ miejscach (login.asp:28, inventory.asp:59-66, CarInventory.vb:50/70/83, SalesOrder.vb:59/78/82)
  • Hasła przechowywane w plain text
  • Hardcoded credentials: sa/Admin@123 w config.asp i VB6
  • Brak HTTPS — cały ruch nieszyfrowany
  • XSS — Session("UserName") bez escapowania
  • Brak ochrony CSRF

Poważne MAJOR

  • Session hijacking (cookie bez HttpOnly/Secure)
  • Brak rate limiting na login
  • Błędy bazy danych wyświetlane użytkownikowi
  • Logowanie do pliku (nie structured logging)
  • SanitizeInput() robi tylko Trim()
  • ValidatePassword() — minimum 6 znaków
Dobra wiadomość: stored procedures są poprawnie sparametryzowane — SQL Injection jest wyłącznie w warstwie ASP/VB6.

Migracja haseł — bez resetu

Dodajemy kolumny PasswordHash + PasswordMigrated. Przy logowaniu walidujemy plain text → haszujemy bcrypt → ustawiamy flagę. Po migracji wszystkich użytkowników usuwamy kolumnę Password.

Rollback w każdej chwili

Legacy system pozostaje uruchomiony przez cały czas migracji. Rollback = zmiana DNS z powrotem. Po 48h oknie monitoringu w tygodniu 14 — legacy archiwizujemy.

Claude Code

Jak używaliśmy Claude Code

Co działało najlepiej

  • Analiza 17 podatności w ASP/VB6 — jednym promptem, z numerami linii
  • Generowanie pełnej dokumentacji migracji (5 plików .md)
  • Tworzenie 60+ test cases na podstawie kodu legacy
  • CLAUDE.md — cały zespół pracuje z jednym spójnym kontekstem
  • Quick Wins — gotowe fragmenty kodu do natychmiastowego zastosowania
  • User Stories zgodne ze standardem INVEST
Największa oszczędność czasu:
Analiza bezpieczeństwa + dokumentacja migracji — szacunkowo 2–3 dni pracy analitycznej w ciągu kilku godzin

Co nas zaskoczyło

  • Rozumienie kontekstu VB6 i Classic ASP bez dodatkowych wyjaśnień
  • Inicjatywa w wykrywaniu podatności niewidocznych na pierwszy rzut oka
  • Spójność między dokumentami — plan migracji odnosi się do konk. linii kodu
  • Jakość test cases — od razu pokrywają edge cases (SQL injection, SESS-07)
Wyzwania:
Weryfikacja czy wygenerowany kod nie zawiera subtelnych błędów logicznych — wymagała ręcznego review każdego fragmentu
